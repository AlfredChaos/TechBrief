# PRD：TechBrief——官方技术博客监控 → 全文翻译 → 公众号草稿生成（MVP：自动草稿，人工发布）

项目名：TechBrief  
版本：v0.1  
状态：草案（可评审）  
负责人（PM）：SOLO（代）  
目标用户：单人运营者  
更新时间：2026-05-22  

---

## 1. 背景与问题陈述

OpenAI 与 Anthropic 等 AI 厂商的官方技术博客更新频繁、信息密度高，但对于中文读者存在语言门槛与获取成本。个人运营者希望：

1) **第一时间发现**新文章；  
2) **稳定抽取正文并全文翻译成中文**，保持结构与代码块完整；  
3) 自动生成符合微信公众号图文规范的内容，**进入草稿箱**；  
4) 由运营者在公众号后台**人工确认后发布**，降低合规与平台风控风险；  
5) 在文末固定追加**来源/作者/原文链接**等信息。

---

## 2. 产品目标（Goals）

### 2.1 业务目标
- 降低“发现—翻译—排版—入草稿”的总耗时，将运营者的人力投入聚焦在“复核与发布”。
- 保持内容结构清晰、术语一致、代码与命令行不被破坏，达到可发布质量。

### 2.2 用户目标
- 用户不需要每天手动巡检官网博客；系统能自动发现并在短时间内生成草稿。
- 用户能清晰知道每篇文章处理进度、失败原因，并可一键重试。

### 2.3 成功指标（MVP）
- 发现新文章 → 草稿生成成功率 ≥ 95%（以可访问页面为前提）。
- 处理时延：
  - **发现时延**：新文章发布后，系统在 **2 小时**内完成一次处理并生成草稿（由固定调度频率决定）。
  - **单次处理耗时**：单次执行中，从拉取源数据到草稿可用的 P50 ≤ 30 分钟，P95 ≤ 2 小时（受翻译与图片处理影响）。
- 结构保真：标题层级、列表、引用、代码块不丢失（抽样验收通过率 ≥ 98%）。

---

## 3. 非目标（Non-goals）

- 不做“全自动发布/群发”（MVP 不接管最终发布动作）。
- 不做多来源扩展（MVP 仅支持 OpenAI 与 Anthropic 官方博客；后续可扩展）。
- 不做深度内容再创作（MVP 为**全文翻译**，可选译者注/摘要作为增强项）。
- 不承诺消除版权与合规风险（仅提供辅助能力与风险提示，最终发布由用户决定）。

## 4. 用户画像与使用场景

### 4.1 用户画像：单人运营者
- 具备基础技术理解，希望快速把英文官方信息同步给中文读者。
- 不希望维护复杂系统；愿意接受轻量后台与少量配置。

### 4.2 核心场景
1) 用户配置来源（OpenAI/Anthropic）与公众号信息 → 系统开始巡检。  
2) 官网发布新文章 → 系统发现 → 抓取 → 抽取 → 翻译 → 排版 → 生成公众号草稿。  
3) 用户打开公众号后台草稿箱 → 阅读与修改 → 人工发布。  
4) 文章抓取/翻译失败 → 用户看到原因 → 一键重试或手动处理。

## 5. 竞品/替代方案（简要）

- 手工：收藏 RSS/邮件订阅 + 复制到翻译工具 + 公众号编辑器排版。成本高、易漏。
- 通用内容同步工具：往往缺少“代码块保护/术语表/公众号排版/可恢复流水线”。

## 6. 需求范围与优先级

### 6.1 P0（MVP 必须）
1) 文章发现：RSS 优先；无 RSS 时轮询列表页（支持 ETag/Last-Modified）。  
2) 去重：按 canonical_url / url hash 去重，避免重复创建草稿。  
3) 文章抓取：获取原始 HTML 并落库（用于追溯与重跑）。  
4) 正文抽取：结构化提取（标题/段落/列表/引用/代码块/图片）。  
5) 全文翻译：  
   - 代码块与命令行不翻译；  
   - 术语一致（维护术语表）；  
   - 保持原有段落结构与链接。  
6) 公众号排版：将结构化内容转为公众号富文本 HTML；处理图片（下载→上传→替换链接）。  
7) 草稿创建：通过公众号接口创建草稿（draft），并记录 draft_id。  
8) 状态与可观测：每篇文章有处理状态、耗时、失败原因；支持重试。  
9) 文末固定追加：来源、作者、原文链接、发布时间（如可获取）、免责声明模板。

### 6.2 P1（增强）
- 运营后台：文章列表、状态筛选、日志查看、手动触发“重跑某一阶段”（例如仅重翻译）。  
- 质量检查：结构差异检测、链接检查、图片缺失检查。  
- 内容策略：允许只翻译部分栏目/标签、黑白名单。  
- 通知：草稿生成成功/失败的邮件/企业微信/Telegram 通知（可选）。

### 6.3 P2（后续）
- 人工审批页 + 点击后自动发布（半自动增强版）。  
- 更多来源站点与自定义 RSS。  
- 摘要/要点提炼、译者注自动生成、SEO 标题优化。

## 7. 端到端流程（MVP）

### 7.1 正常流程
1) Scheduler 触发巡检（**每 2 小时一次**，固定调度）。  
2) Discover：拉取 RSS 或列表页 → 新 URL 入队列。  
3) Fetch：抓取详情页 HTML → 存 `raw_html`。  
4) Extract：抽取正文与元信息 → 生成 `content_ast` / `content_md`。  
5) Translate：按块翻译 → 生成 `zh_ast` / `zh_md`。  
6) FormatForWeChat：AST → HTML；图片处理（下载→上传微信→替换 URL）。  
7) CreateDraft：调用公众号草稿接口创建草稿 → 返回 `wechat_draft_id`。  
8) 结束：状态置为 `drafted`，等待用户去公众号后台发布。

### 7.2 异常与降级策略
- RSS 不可用：降级为列表页轮询。  
- 抽取失败：保存 raw_html，标记失败原因，允许人工指定 CSS 选择器/规则（P1）。  
- 翻译失败：支持重试（指数退避）；保留已翻译段落的 checkpoint。  
- 图片下载/上传失败：  
  - 允许跳过单张图片并标记占位（默认：失败即阻断，避免生成残缺草稿）；  
  - P1 增加“忽略图片失败并继续”的开关。  
- 公众号接口失败：记录错误码与响应；自动刷新 token 并重试一次；仍失败则进入 `failed`。

## 8. 功能需求（详细）

### 8.1 来源管理
**FR-001** 支持来源：OpenAI Blog、Anthropic Blog（MVP 固定内置）。  
**FR-002** 对每个来源支持配置：
- 启用/停用  
- 巡检频率：**MVP 固定为每 2 小时一次**（不提供频率配置项）  
- 栏目/标签过滤（P1）  
**FR-003** 记录每次巡检结果与增量数量。

### 8.2 文章发现与去重
**FR-010** 新文章判定：
- 以 canonical_url 为主；无 canonical 则用最终 URL；  
- dedupe_key = sha256(source + canonical_url)。  
**FR-011** 已处理文章不重复入队；允许“强制重跑”（P1）。  

### 8.3 抓取与抽取
**FR-020** 抓取：
- 支持条件请求（ETag/Last-Modified）。  
- 保存 raw_html（含抓取时间、HTTP 状态、headers 摘要）。  
**FR-021** 抽取：
- 结构单元：H1-H4、P、UL/OL、BLOCKQUOTE、CODE_BLOCK、IMAGE、HR。  
- 保留原文链接；图片保留源 URL。  
**FR-022** 元信息解析：
- title、author（若页面可得）、published_at（若页面可得）、cover_image（若可得）。

### 8.4 翻译与术语一致
**FR-030** 翻译策略：
- code block/inline code 原样保留；  
- 链接 URL 不翻译，链接文本可翻译；  
- 保留段落与标题层级；  
- 可配置目标中文风格：技术博客直译风/更通顺意译风（默认：直译风）。  
**FR-031** 术语表：
- 支持维护 glossary（YAML/JSON），形如 `{"tool use":"工具调用","token":"令牌/Token(保留)"}`；  
- 翻译前后执行一致性替换（注意边界与大小写）。  
**FR-032** 质量检查（MVP 最小）：
- 标题数量与层级差异检测；  
- code block 数量一致性检测；  
- 关键链接数量一致性检测。

### 8.5 公众号排版与草稿
**FR-040** HTML 生成：
- 兼容公众号常见渲染限制：尽量使用简洁标签（p/h2/h3/strong/em/pre/code/img/blockquote/ul/ol/li）。  
- 表格降级策略（若原文含 table）：转换为图片或改为列表（P1 优化）。  
**FR-041** 图片处理：
- 下载到临时存储 → 上传微信素材/获取可用 URL → 替换正文中的 img src。  
- 图片去重（按内容 hash）。  
**FR-042** 草稿创建：
- 创建图文草稿（标题、作者字段、封面图、正文）；  
- 保存 wechat_draft_id；  
- 若失败，记录错误码与原始响应摘要。

### 8.6 文末署名与声明
**FR-050** 文末自动追加固定区块（模板可配置），至少包含：
- 来源站点名称  
- 原文标题（可选）  
- 作者（若可获取；不可获取则省略或标记“未标注”）  
- 原文链接（必填）  
- 发布时间（若可获取）  
- 版权/免责声明（可配置）

### 8.7 运营后台（P1）
**FR-060** 列表页：按状态筛选（discovered/fetched/translated/drafted/failed）。  
**FR-061** 详情页：展示原文/译文/渲染预览、错误日志、重试按钮。  
**FR-062** 重跑能力：从指定节点重跑（例如仅重翻译/仅重排版）。

## 9. 权限、合规与风控（产品约束）

### 9.1 版权/条款风险
系统提供“署名/链接/来源声明”能力，但**不保证获得转载授权**。建议在产品中：
- 首次配置时提示用户确认来源站点条款；  
- 提供“仅摘要 + 原文链接”（P1 可选策略）作为风险更低的替代。

### 9.2 公众号风控
即使进入草稿箱，也可能因内容触发审核或限制。MVP 保持“人工发布”能显著降低误发风险。

## 11. 技术方案（与 PRD 相关的约束/接口）

> PRD 不锁死实现，但给出可行边界，便于估算与验收。

- 编排：LangGraph（状态机 + checkpoint + 节点可重试）。  
- 存储：SQLite（MVP）→ Postgres（增强/多用户）。raw_html/图片可用对象存储（P1）。  
- 抽取：Readability/Trafilatura 类正文抽取；必要时自定义规则。  
- 翻译：可配置 LLM 提供方；需支持术语表与代码块保护。  
- 微信接口：使用公众号官方 API（草稿/素材/上传）；token 需要安全存储与刷新机制。

## 14. 验收标准（MVP）

1) 对 OpenAI 与 Anthropic 各选 5 篇历史文章回放处理：  
   - 10/10 成功生成草稿（允许 1 次手动重试）；  
   - 标题层级、代码块、主要图片不丢失。  
2) 新文章发现：在模拟新增（或实际新增）情况下，系统能在配置周期内发现并入库。  
3) 去重：同一文章重复巡检不会重复创建草稿。  
4) 可追溯：任一失败能看到失败阶段与错误摘要；能从失败点重试并成功。  
5) 文末署名模板在草稿中正确出现，包含来源/作者（如有）/原文链接。  

---

## 15. 风险与对策

- **来源站点结构变化** → 抽取失败：保留 raw_html + 规则可配置 + 报警。  
- **翻译质量波动**：术语表 + 结构校验 + 人工发布兜底。  
- **公众号接口限制/变更**：将“草稿创建/图片上传”封装为独立适配层，便于替换。  
- **合规/版权不确定**：产品提示 + 可选摘要模式（后续）。
